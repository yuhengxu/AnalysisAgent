## 4. 后端开发规范

### 4.1 API 层

API 层只负责：

- 解析请求参数。
- 注入数据库 Session。
- 注入当前用户。
- 调用 service。
- 把 ValueError 转成 HTTPException。
- 返回响应。

不要在 API 层写复杂业务逻辑。

### 4.2 Service 层

Service 层负责：

- 数据库查询。
- 数据库写入。
- 文件导出。
- 图表生成。
- 调用 Skill。
- 数据隔离。
- 业务异常。

### 4.3 Skill 层

Skill 层负责：

- 证据采集。
- prompt 构造。
- LLM 调用。
- 输出结构化 JSON。
- normalize。
- fallback。

LLM 调用必须包裹降级逻辑，不能让外部模型失败导致业务接口 500。

### 4.4 Model 层

Model 层只定义表结构，不写业务逻辑。

新增模型后必须在：

```text
backend/app/models/__init__.py
```

中 import 并加入 `__all__`。

### 4.5 权限

新增页面或模块时必须同时做两层权限：

1. 前端 `PageRoute` / `AdminRoute`。
2. 后端 `require_page` / `require_admin`。

不能只做前端隐藏。

## 5. 前端开发规范

### 5.1 API 调用

新增后端接口后，先在：

```text
frontend/src/api/client.ts
```

中增加调用函数，再在页面使用。

### 5.2 页面权限

新增业务页面：

1. 在 `App.tsx` 用 `PageRoute page="xxx"` 包裹。
2. 在 `Layout.tsx` 的 `navItems` 中增加 `{ key, to, label }`。
3. 在后端 `models/user.py` 增加页面 key 和 label。
4. 在后端对应 router 加 `require_page("xxx")`。
5. 更新 `docs/17-用户管理与权限.md`。

新增管理员页面：

1. 在 `App.tsx` 用 `AdminRoute` 包裹。
2. 在 `Layout.tsx` 的 `adminNavItems` 中增加入口。
3. 后端使用 `require_admin`。

### 5.3 状态管理

当前使用组件级 `useState/useEffect`，没有全局状态库。

全局登录态在：

```text
contexts/AuthContext.tsx
```

## 6. 数据隔离规范

任何用户私有资源都必须满足：

```text
创建时写入 user_id
查询时 WHERE user_id = 当前用户.id
详情、更新、删除、导出、修订全部复用同一个 get 方法
越权返回 404
```

当前已经严格隔离：

- reports
- predictions
- report tasks
- prediction tasks

后续新增私有资源时必须照此处理。

## 7. 异步任务规范

当前异步任务文件：

```text
services/report_tasks.py
services/prediction_tasks.py
```

规范：

1. 创建任务时写入 `user_id`。
2. 轮询时校验 `task.user_id == 当前用户.id`。
3. 后台线程创建自己的 `SessionLocal()`。
4. finally 中关闭 Session。
5. 当前只能单 worker。
6. 多 worker 需要迁移到 DB / Redis。

## 9. 自检清单

后端：

```bash
cd backend
python -c "from app.main import app"
```

前端：

```bash
cd frontend
npm run build
```

手工权限检查：

```text
1. admin 登录。
2. 创建 user。
3. 给 user 只授权 reports。
4. user 登录。
5. 导航只显示 reports。
6. user 访问 /data 显示无权访问。
7. user 直接请求 /api/v1/data/datasets 返回 403。
```

数据隔离检查：

```text
1. 用户 A 生成月报。
2. 用户 B 生成月报。
3. 用户 A 列表只看到自己的月报。
4. 用户 A 访问用户 B 的 /reports/{id} 返回 404。
5. 用户 A 导出用户 B 的报告返回 404。
6. 预测分析表同理。
```

任务隔离检查：

```text
1. 用户 A 创建 report task。
2. 用户 B 轮询该 task_id 返回 404。
3. 用户 A 能正常轮询并得到结果。
```

## 10. 当前已知技术债

1. 未使用 Alembic，表结构演进依赖轻量迁移。
2. 异步任务状态在内存中，不支持多 worker。
3. SQLite 适合单机轻量部署，高并发场景建议迁移 PostgreSQL。
4. 图1-2/图1-3 持仓图需要 CFTC/ICE 数据源补齐。
5. 预测模型仍是占位级 MA3 + 趋势模型。
6. 前端首包较大，可做路由懒加载。
7. 需要补充自动化测试。
8. 需要用户自助改密和首次登录强制改密。
