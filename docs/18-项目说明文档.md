# AnalysisAgent 项目说明文档

## 1. 项目定位

AnalysisAgent 是一个面向能源行业，尤其是国际原油领域的 AI 数据分析平台。系统将行业数据导入、统计分析、可视化、大模型研判、预测分析表生成、国际油价月报生成、在线人工校准与文件导出串成一套可复用工作流。

平台的两个核心交付物是：

1. **油价预测分析表**：围绕 6 大类、28 项影响因素，生成重要性、形势判断、对油价影响、价格区间预测，并导出 Excel。
2. **国际油价月报**：围绕期货市场、现货市场、宏观、需求、供应、库存、地缘、持仓、油价展望等章节生成结构化月报，并导出 Word / PDF / LaTeX。

Agent 是统一入口，不是外挂功能。用户既可以在各业务页面操作，也可以通过右下角粒子化 Logo Agent 发起数据分析、预测分析表生成、月报生成、章节修订等任务。

### 2.1 技术栈

| 层级 | 技术 |
|---|---|
| 前端 | React 18、TypeScript、Vite、TailwindCSS、React Router、ECharts、tsparticles |
| 后端 | Python 3.12、FastAPI、SQLAlchemy 2.0、Pydantic v2 |

| 文件导出 | openpyxl、python-docx、LaTeX / xelatex / pandoc |
| 图表导出 | matplotlib、PyEcharts、Playwright |

| 存储 | SQLite + 本地文件系统 |
| 部署 | Docker Compose；前端 Nginx + 后端 Uvicorn |

### 2.2 分层架构

```text
前端 frontend/src
  ├─ pages/             页面：Dashboard、DataCenter、Prediction、ReportCenter、UserManagement 等
  ├─ components/        组件：Layout、PageRoute、AdminRoute、ChartPanel、FloatingAgent 等
  ├─ api/client.ts      前端 API 调用层
  └─ contexts/          AuthContext 登录态与权限状态

        │ HTTP /api/v1/*

后端 backend/app
  ├─ api/               FastAPI 路由层：只做请求解析、依赖注入、异常转换
  ├─ services/          业务服务层：DB 读写、数据聚合、文件导出、任务编排
  ├─ skills/            智能产出层：证据采集 + LLM 调用 + 结构化产出 + 规则兜底
  ├─ models/            SQLAlchemy ORM 表模型
  ├─ templates/         预测表、月报、LaTeX、样例契约等结构模板
  ├─ schemas/           Pydantic 请求体
  └─ core/              基础设施：配置、数据库、鉴权、LLM、日志

        │ ORM / 文件 IO

SQLite db/energy_platform.db
data/raw/       原始上传文件
data/exports/   Excel / Word / PDF / TeX 导出结果
data/charts/    月报图表 PNG
logs/           运行日志
```

### 2.3 核心设计原则

1. **API 层只做编排**：解析请求、注入 Session、调用 Service、把异常转换成 HTTP 状态码。
2. **Service 层负责业务**：数据库读写、统计聚合、文件生成、图表生成、调用 Skill。
3. **Skill 层负责智能产出**：采集证据、调用大模型、结构化输出、失败时规则兜底。
4. **结构化内容入库**：预测分析表和月报都以 `content_json` 存库，前端编辑和后端导出共用同一结构。
5. **LLM 必须可降级**：模型不可用时，系统不能 500，必须走数据库证据 + 规则兜底。
6. **系统表 snapshot 只读**：月报 6 张动态系统表由数据中心预置到 `report_table_snapshots`，生成、打开、导出时只读加载，LLM 不填表格 rows。
7. **多用户严格隔离**：月报、预测分析表、异步任务都按 `user_id` 隔离，管理员也不能默认查看其他用户数据。
8. **文档驱动开发**：`README.MD` 是项目宪法，`GUIDE.md` 是工作入口，`docs/` 是模块设计权威文档，`plans/` 是实施方案。

## 3. 后端整体框架

### 3.1 应用入口

后端入口为：

```text
backend/app/main.py
```

启动流程：

```text
setup_logging()
  → create_app()
      → 配置 CORS
      → 注册请求耗时日志中间件
      → 注册 auth / users / data / analysis / analytics / agent / reports / forecast / prediction / llm_logs / llm 路由
      → startup:
          → init_db()
          → run_migrations()
          → ensure_default_admin()
          → ensure_default_template()
      → 暴露 /health
```

### 3.2 数据库

数据库使用 SQLite，默认文件：

```text
db/energy_platform.db
```

核心表：

| 表 | 说明 |
|---|---|
| users | 用户、角色、页面权限、启停状态 |
| datasets | 导入数据集元信息 |
| price_series | 原油价格时间序列 |
| balance_forecasts | 机构供需预测 |
| factor_assessments | 历史影响因素评估 |
| forecast_results | 统计预测模型结果 |
| predictions | 预测分析表实例 |
| reports | 月报实例 |
| report_templates | 月报模板 |
| report_table_snapshots | 月报 6 张动态系统表快照 |
| agent_runs | Agent 调用审计 |
| llm_dialogue_logs | 大模型调用日志 |

### 3.3 轻量迁移

当前项目未使用 Alembic，而是在启动时通过 `core/migrate.py` 做轻量补列，包括：

- `reports.user_id`
- `predictions.user_id`
- `users.allowed_pages_json`
- `users.is_active`
- `users.updated_at`
- `report_table_snapshots`
- `agency_forecast_manual`

后续生产化建议引入 Alembic。

### 3.4 SQLite 并发

当前 SQLite 已启用：

```text
PRAGMA journal_mode=WAL
PRAGMA synchronous=NORMAL
PRAGMA busy_timeout=30000
connect_args timeout=30
pool_pre_ping=True
```

这可以支持单机多用户并发读写，但异步任务状态仍保存在进程内存中，因此生产部署必须使用单 worker。

## 4. 前端整体框架

### 4.1 页面结构

| 页面 | 路径 | 权限 | 说明 |
|---|---|---|---|
| Login | `/login` | 公开 | 登录页 |
| Dashboard | `/` | dashboard | 总览、核心指标、快捷入口 |
| DataCenter | `/data` | data | 数据导入、数据浏览、质量检查、月报表数据 |
| Analysis | `/analysis` | analysis | 智能分析与图表 |
| Prediction | `/prediction` | prediction | 预测分析表生成、编辑、导出 |
| Forecast | `/forecast` | forecast | 统计预测模型与回测 |
| ReportCenter | `/reports` | reports | 月报生成、编辑、修订、导出 |
| UserManagement | `/users` | admin | 用户管理 |
| Settings | `/settings` | admin | 大模型设置 |
| LlmMonitor | `/monitor` | admin | LLM 调用日志 |

### 4.2 权限路由

前端有三类路由保护：

```text
ProtectedRoute  未登录重定向到 /login
PageRoute       根据 allowed_pages 判断普通页面访问权
AdminRoute      限制 admin 页面
```

导航栏 `Layout.tsx` 会基于 `canAccessPage(page)` 过滤普通用户可见导航项，管理员额外显示“用户管理 / 系统设置 / 监控日志”。

### 4.3 API 调用层

所有前端请求集中在：

```text
frontend/src/api/client.ts
```

特点：

- `baseURL=/api/v1`
- token 存在 localStorage
- 请求自动附加 `Authorization: Bearer <token>`
- 401 自动清理 token 并跳转登录页
- 文件下载使用 `responseType: 'blob'`
- 轮询任务使用短 timeout 的 `pollApi`
- 生成、修订、Agent 请求自动附加大模型 provider / model / mode 参数

## 5. 用户与权限设计

### 5.1 用户角色

| 角色 | 说明 |
|---|---|
| admin | 管理员，可进入用户管理、系统设置、监控日志 |
| user | 普通用户，只能进入管理员授权的页面 |

### 5.2 页面权限

普通用户可分配页面：

```text
dashboard
data
analysis
prediction
forecast
reports
```

管理员专属页面：

```text
users
settings
monitor
```

### 5.3 后端权限

核心依赖：

```text
get_current_user   解析 JWT，校验用户存在且启用
require_admin      要求 role == admin
require_page       要求普通用户拥有页面权限；admin 放行
```

路由权限：

| API 前缀 | 权限 |
|---|---|
| `/auth/login` | 公开 |
| `/auth/me` | 登录 |
| `/users/*` | admin |
| `/data/*` | data |
| `/analysis/*` | analysis |
| `/forecast/*` | forecast |
| `/prediction/*` | prediction |
| `/reports/*` | reports |
| `/analytics/dashboard` | dashboard |
| `/analytics/*` 其他接口 | 登录 |
| `/llm/*` | admin |
| `/llm-logs/*` | admin |

### 5.4 数据隔离

以下资源按 `user_id` 严格隔离：

- 月报 reports
- 预测分析表 predictions
- 月报异步任务 report tasks
- 预测异步任务 prediction tasks

规则：

```text
user_id == 当前登录用户.id
```

越权访问返回 404，避免泄露资源是否存在。管理员也不能默认查看其他用户的月报或预测表。

## 6. 业务流程

### 6.1 首次启动流程

```text
启动后端
  → 初始化日志
  → 建表
  → 运行轻量迁移
  → 确保默认 admin 存在
  → 把历史 user_id=NULL 的月报归属默认 admin
  → 确保默认月报模板存在
  → 服务可用
```

### 6.2 数据导入流程

```text
用户进入数据中心
  → 上传 CSV/XLSX 或导入 yuebao 样例
  → 后端保存原始文件到 data/raw/
  → 自动识别 category
  → 解析价格、供需、影响因素
  → 写入 SQLite
  → 做质量检查
  → 若是月报相关数据，自动同步部分派生表
  → 前端刷新数据集与图表
```

### 6.3 月报表数据预置流程

```text
用户进入数据中心 → 月报表数据
  → 选择回顾月
  → 同步派生数据
      表1-1 价格月度变化
      表2-3 供需差
      表3-1 情景预测
  → 深度研究联网获取 / 手工填写 web 表
      表2-1 PMI
      表2-2 GDP
      表3-2 机构预测
  → 保存到 report_table_snapshots
```

### 6.4 预测分析表生成流程

```text
用户进入预测分析表页面
  → 选择年份、月份、品种、补充说明
  → POST /prediction/generate 或 /prediction/generate/async
  → PredictionService.generate()
  → PredictionSkill.gather_evidence()
      读取价格统计、价差、供需、历史因素
  → 调用 LLM / DeepSearch
  → normalize 成固定 28 项结构
  → 写入 predictions(user_id=当前用户)
  → 前端展示可编辑表格
  → 用户人工校准
  → 保存
  → 导出 Excel
```

### 6.5 月报生成流程

```text
用户进入报告中心
  → 选择展望月
  → 前端计算 review_month / outlook_month
  → 检查 6 张系统表是否预置
  → POST /reports/generate 或 /reports/generate/async
  → ReportService.generate_monthly_draft()
  → ReportSkill.gather_evidence()
  → LLM 撰写摘要和章节正文
  → ReportTableDataService.load_for_report()
      从 report_table_snapshots 注入 6 张系统表
  → guard_report_content()
  → 写入 reports(user_id=当前用户)
  → 生成图表 PNG
  → 导出 Word
  → 前端展示可编辑月报
  → 用户章节修订 / Agent 校准
  → 保存
  → 导出 Word / PDF / TeX
```

### 6.6 Agent 对话流程

```text
用户点击右下角粒子化 Agent
  → 输入自然语言
  → POST /agent/chat
  → AgentService 判断是否为视觉指令
      是：返回 visual 状态
      否：识别 skill
  → 调用 prediction/report/forecast/analysis/web_search 等能力
  → 记录 agent_runs
  → 返回 message、visual、charts、report_id、prediction_id
  → 前端可跳转到对应报告或预测表
```

## 8. 部署与运行

### 8.1 Docker Compose

```bash
docker compose up --build
```

访问：

```text
前端：http://localhost:3000
后端：http://localhost:8000
健康检查：http://localhost:8000/health
```

### 8.2 本地开发

后端：

```bash
cd backend
pip install -r requirements.txt
cp .env.example .env
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

前端：

```bash
cd frontend
npm install
npm run dev
```

访问：

```text
http://localhost:5173
```

### 8.3 生产约束

当前异步任务状态保存在后端进程内存字典中，因此生产必须单 worker：

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 1
```

不要直接使用：

```bash
uvicorn app.main:app --workers 4
```

否则可能出现 worker A 创建任务、worker B 轮询任务导致 404 的问题。

如需真正多 worker，需要把任务状态迁移到数据库或 Redis。

## 9. 运维与排障

### 9.1 日志

日志位置：

```text
logs/app.log
```

主要 logger：

| logger | 说明 |
|---|---|
| api | HTTP 请求、启动日志 |
| llm | 大模型调用 |
| skill.prediction | 预测 skill 降级 |
| skill.report | 月报 skill 降级 |
| app | 日志系统自身 |

### 9.2 常见问题

| 现象 | 排查 |
|---|---|
| 登录失败 | 检查 admin 初始账号、密码、JWT_SECRET_KEY |
| 账号停用 | 管理员在用户管理页启用账号 |
| 页面无权访问 | 检查用户 allowed_pages |
| API 返回 403 | 后端 require_page 生效，需授权页面 |
| 月报/预测详情 404 | 资源不属于当前用户，或资源不存在 |
| 异步任务轮询 404 | task_id 不属于当前用户，或多 worker 部署导致内存任务丢失 |
| 大模型不生效 | 检查 API Key、DEFAULT_LLM_PROVIDER、DeepSearch Bot ID |
| 月报系统表缺失 | 先到数据中心「月报表数据」预置 6 张系统表 |
| SQLite locked | 检查是否高并发写入；当前已启用 WAL 和 busy_timeout |

## 10. 后续增强建议

1. 引入 Alembic，替代启动时轻量补列。
2. 把异步任务状态迁移到数据库或 Redis，支持多 worker。
3. 增加用户自助改密和强制首次登录改密。
4. 增加管理员审计权限，但不要默认放开跨用户月报查看。
5. 增加自动化测试，包括权限、数据隔离、导出、异步任务。
6. 增加结构化日志、trace_id、Prometheus 或 Sentry。
7. 对前端做路由级懒加载，降低首包体积。
